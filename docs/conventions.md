# Okane 开发规范与准则 (Conventions)

## 1. 核心设计哲学

### 1.1 DDD (领域驱动设计)
- **业务核心化**：业务逻辑必须高度内聚在 `core` 领域层，基础设施层（Database, Network）仅作为适配器（Adapter）存在。
- **实体显式化**：所有的业务状态变迁必须通过领域实体或聚合根的方法进行，严禁破坏封装性。

### 1.2 Rust 风格 (Idiomatic Rust)
- **类型安全**：利用 Rust 强类型系统（如 NewType 模式）在编译期消除业务错误。
- **所有权明确**：严格管理 `Arc` 与 `Weak` 的引用循环，确保资源能及时回收。

## 2. 质量红线与禁令

### 2.1 严禁玩具编程 (No Toy Programming)
- 拒绝任何为了快速上线而做的简陋实现（Garbage Output）。
- 代码必须具备生产级的健壮性，涵盖错误边界与边缘情况。

### 2.2 严禁基于假设编程 (No Assumption-Based Coding)
- **禁止猜测**：逻辑必须有明确的官方文档出处或严密的数学证明。
- **案例规避**：严禁出现“假设取 3 倍长度以保证收敛”这种缺乏依据的 Hard-coded 假设。

### 2.3 禁止简易实现 (No Oversimplified Implementation)
- 严禁为了逃避复杂性而编写不可扩展的逻辑。
- 每一步产出的代码都必须是高质量、经得起推敲、可长期维护的。

## 3. 交易系统专项规范

### 3.1 财务计算安全性
- **禁止浮点数**：金额、价格、成交量计算**必须**使用 `rust_decimal::Decimal`。
- **严禁强制转换**：任何可能导致精度丢失的 `as` 转换都必须经过严格审查，优先使用 `.into()` 或 `.try_into()`。

### 3.2 错误处理 (Error Handling)
- **严禁 unwrap**：除非在逻辑上可以 100% 证明为 `unreachable` 路径。
- **错误透明**：所有的 IO、业务错误必须被包装成具有语义的自定义 Error 类型（如 `TradeError`, `MarketError`）。
- **语言规范**：错误处理字符串**必须**使用英文。严禁使用 emoji 或其他非英文字符，以确保字体兼容性。
- **格式规范**：Error 信息必须全部使用**小写字母**（小写），除非是专业缩写（如 `OMS`, `API`）。

### 3.3 日志输出 (Logging)
- **语言规范**：日志输出内容**必须**使用英文。严禁使用 emoji 或其他语言。
- **格式规范**：Log 信息可以使用大写字母。

## 4. 编码硬性约束

- **单模块单目录**：每个业务逻辑在 `crates/` 下应有独立的目录体系，严禁出现数千行的“上帝文件”。
- **lib.rs 零逻辑**：`lib.rs` 仅用于模块声明 (`mod`) 与必要的导出，不应包含具体的业务逻辑代码。
- **强制注释规范**：对外暴露的函数方法必须包含 `///` 文档注释：
    - `# Logic`: 核心逻辑步骤描述。
    - `# Arguments`: 参数含义及其约束。
    - `# Returns`: 返回值含义及可能的 Error 场景。

## 5. 工具链配置

- **Clippy**: 必须通过工作区定义的严格等级检查。
- **Rustfmt**: 统一使用默认配置进行格式化。
- **Coverage**: 使用 `cargo llvm-cov` 作为覆盖率基准工具。

## 6. 测试标准 (Testing Standards)

- **基础框架**：以 `cargo test` 为核心。
- **覆盖率基准**：
    - **核心要求**：除 `app` 外，所有的子 crates 必须达到 **80% 以上** 的代码覆盖率（LLVM Coverage）。其中，`api` 的测试覆盖率要求达到 **100%**。外部依赖库可酌情降低覆盖率。
    - **严谨性**：严禁通过欺骗、虚假测试或无意义的断言来刷覆盖率。
- **用例要求**：
    - 必须编写严谨有效、逻辑覆盖完整的单元测试（Unit Test）或集成测试（Integration Test）。
    - 禁止编写重复的测试代码，禁止在一个测试函数里编写多个测试逻辑。
    - **严禁日志输出**：测试代码中严禁输出任何 Log（控制台输出或文件日志）。
    - **断言驱动**：必须使用断言（Assertions）来判断程序逻辑是否正确，而非通过观察日志。
- **可维护性**：测试用例应具备良好的可读性，确保在业务逻辑变迁时能及时发现回归。
